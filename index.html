Description

OLED battle is a multiplayer game inspired by the popular game pong. The gameplay consists of two players controlling rectangular paddles on adjacent oled screens. Each screen represents one half of the game court, and presents one paddle that players may move horizontally using the accelerometer in a connected CC3200 Launchpad. In Addition to this players may press SW2 to serve a ball and make themselves invulnerable for a short duration by pressing SW3. Balls bounce off the side of the court, other balls, and player paddles while they are invulnerable. The game ends when a player is struck by a ball.

In addition to gameplay, OLED battle allows players to select a number of matches to play with a settings selection screen, presents players with their scores when a player loses a match, and sends player scores to them via Amazon Web Services when the game has completed.

Design

Below we see the state machine for our application without going into gameplay details. We see that players initially view a start screen, and after any button press are navigated to a settings screen where they can modify settings. Following the settings screen, players play a number of game they have selected, and after that their scores are sent and they view a game over screen with the possibility of replaying their matches.

Below we see a state diagram for the gameplay of OLED battle. When the games begin, paddles are displayed on the screen and interrupt handlers are registered and enabled. After this, the game enters a loop where it updates the player’s location and status followed by the ball or bullets locations until a player is struck by a ball.

Below we have our system architecture view. In our architecture, user’s interface with their own launchpad microcontroller using button presses on the controller, sending signals with a remote that is picked up by an infrared sensor, and by tilting the controller. Players receive feedback from the devices mainly through the OLED’s display, but also through amazon’s IoT. In addition to this, the two launchpads communicate with each other by sending messages over UART.

Implementation

Below is the circuit schematic for our project. Our microcontroller interfaces with the OLED screen and the IR receiver, both off-board peripherals. It also communicates with the other player’s microcontroller using UART.

The program starts by initializing the board, all the relevant pins, SysTick, the protocols SPI and I2C, and finally connects to the AWS server. We used the same initialization settings for SPI and I2C from previous labs since we aren’t using any new sensors. We configured our microcontroller to FIFO mode for UART and set its level to 4 characters. All of our transmissions fit within under 4 characters; transmission under 4 characters are padded.

The start screen is the first thing the players see when they start or restart the game. It is a static screen that is coded to respond to any signal from the TV remote.

After leaving the start screen, the program enters the settings screen. While on this screen, the program responds to signals from the TV remote by decoding it into bits, storing it into an array, and passing it through a switch case statement to check the remote’s key presses.

The only two key presses the program responds to are 2, 8, and 5, which we designated as up, down, and confirm respectively. If 2 and 8 are pressed, a variable indicating the position of the cursor is incremented or decremented.

The program behaves differently depending on where the cursor is when the key 5 is pressed: if it is hovering over the start button, it initiates the game, otherwise it sets a global variable reflecting the number of matches to be played from the displayed option.

When the game starts, the program initializes global variables for the player’s state, the player’s score, a struct list containing all bullets on the player’s screen, and integer tracking the number of bullets on screen, and the current number of rounds played. It enters a while loop that only breaks when either player has been struck by a bullet. In every iteration of this loop, the program reads in values from the on-board sensors: the accelerometer and the two push buttons located on the bottom of the board.

The accelerometer modifies the player paddle character’s x-velocity, allowing it to move along the x-direction at a set y-position.

The paddle has three possible states: ready-to-fire, invulnerability, and default. Pushing the right button on the microcontrollers takes the player from ready-to-fire to default. Pushing the left button changes the state from default to invulnerable.

When a player’s state changes from ready-to-fire to default, a bullet is added on the top of their screen at the same x-position as the paddle. It travels towards the player at the default vertical velocity. When the player’s state is invulnerable, the paddle deflects the bullet when it collides with the paddle.

The general behavior for all game objects is that its new position is updated by adding its current position with its x and y velocities. If the new position falls outside the screen’s vertical borders, the program prevents the object from traveling there either by reflecting its velocity in the case of bullets, or simply just setting the new position of the object such that it sits on the edge of the interior border.

If a bullet hits the bottom side of the screen, it will be deleted. If it hits the top side of the screen, it will be deleted, and a UART transmission encoding the bullet’s position and velocity will be sent to the other microcontroller. If a bullet strikes the player’s paddle while it is not invulnerable, the round will end, and a transmission will be sent over UART informing the other microcontroller that the round is over.

Lastly, the game checks for colliding bullets by looping through the bullet list and checking the new position (sum of current position and new velocity) against that of every other bullet. If the positions fall within twice the bullets radius, then a collision has occurred, and both bullets will reverse the sign of their velocity.

When the round ends, the game checks to see if the number of rounds played is equal to the total number of rounds in the game. If it is, the game displays a game over screen, finalizes the game’s score, and sends it to the AWS IoT server, which will then use SNS to send an email to the player with the game’s score. If there are more rounds to be played, the game resets all gameplay variables, and updates the score count, incrementing the player’s score if the game ended via UART transmission, or incrementing the opponent’s score if the game ended through internal logic (ball hitting paddle).

Challenges

We did not encounter many serious challenges in implementing this project. We attribute this to our prior experience working with most of the components of this project.

One challenge that we did encounter was in configuring the pins on our microcontroller so that the various methods of communication would work. We were able to solve this problem by reading the documentation on the launchpad and determining which pins needed to be configured for specific communication methods, and then setting all of the remaining pins that we needed after that.

In addition to this, we had some trouble implementing UART2 communication when sending balls between the boards. We initially had an error where the UART2 interrupt handler would incorrectly be called anytime a UART2 message was sent. We were able to fix this issue by changing our methodology for triggering the UART buffer to be sent.

Finally, we had some issue with Amazon Web Services not functioning properly when our program was flashed, but we were able to resolve this by configuring the program’s network settings.

Future Work

In the future, we would like to improve the synchronization of the two boards by modifying them to have a master/peripheral relationship, and having the two send acknowledgements between each other when players begin or finish a game.

In addition to this, we would also like to implement more settings options on the settings screen. In our current iteration certain gameplay characteristics such as paddle size and ball size are implemented such that the game does not require these to be specific values. Because of this , it would be very straightforward to allow players to modify the paddle dimensions before playing.
